# On-device ML/heavy-resource actor lifecycle: lazy load off init, cache failures, reentrancy-safe shared load task, and grade eviction reason (backgrounded=cancel-now vs memory-pressure=defer-until-idle)

**Extracted:** 2026-07-12
**Context:** Mined from Cubby iOS session logs (0018, 0019); adversarially verified.

## Problem
Two related traps in managing a heavy on-device model (here MLX VLM, but generalizes to any Core ML/large-buffer resource wrapped in an actor): (1) eagerly loading the model at object `init` time hung/crashed on a sibling project (Paperix `DocEnhanceModel.swift`, fixed by commit `1f3040c` = lazy-load off init) — the lesson was explicitly imported into this project's design. (2) Even with lazy loading done right, naively treating every low-memory eviction the same way caused a real production regression: a low-memory warning triggered `vlmEngine.evict()`, which cancelled its own in-flight generation (`CancellationError` after 0.4s), which then cascaded into the downstream refiner timing out because its timeout budget (4s) assumed the primary path wouldn't randomly self-cancel mid-flight. Root cause only surfaced by decoding a real device diagnostic log (memory pressure timestamp lined up exactly with the cancellation).

## Solution
(1) Never eagerly load a heavy model at type/actor init — gate loading behind the actor's first-use call, cache a load failure so repeat calls don't retry a doomed load, and make concurrent callers share one in-flight load `Task` (reentrancy-safe) rather than racing multiple loads. (2) Split eviction into typed reasons with different urgency: a `.backgrounded` reason cancels in-flight work and unloads immediately (the resource — e.g. Metal — genuinely cannot run in the background); a `.memoryPressure` reason must NOT cancel active generation — instead set a pending-evict flag and defer the actual unload until the in-flight work resolves on its own. Loosen any downstream timeout that assumed the primary path never self-interrupts. Track in-flight work per-caller (e.g. UUID-keyed) so a deferred unload waits for the *last* active caller, while a backgrounded unload cancels *all* of them.

## When to Use
This pattern applies to any iOS app running a local heavy resource under an actor — Core ML models, large image/video buffers, on-device speech or vision models — and needing correct behavior under both app backgrounding and iOS memory-pressure notifications, which are semantically different events with different correct responses. The lazy-load-off-init lesson and the graded-eviction-reason distinction are both non-obvious until you've been burned by them, and neither is Cubby- or MLX-specific.
