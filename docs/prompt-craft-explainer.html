The problem

Two frictions, one of them a mirage

The epic named two problems. Arguing from first principles, only one of them was real — and that distinction shaped the entire build.

① Determinism — real

A vague prompt ("fix the thing") routes to whatever the model improvises. No restated goal, no acceptance criteria, no confirm step. The same ask runs differently twice.

  • Scope creep from unstated assumptions
  • Work starts before "done" is defined
  • No artifact a human can correct up front

② Discovery — mostly a mirage

"You have to know a command exists to use it." But Claude Code already injects every skill's description into context and auto-invokes by semantic match — discovery is a solved, built-in problem at 24 commands.

  • A keyword router would duplicate a platform feature
  • …and a weaker version of it (proved later)
  • "As commands grow" is a future problem — YAGNI

So prompt-craft attacks determinism head-on, leans on the platform for discovery, and puts the router behind an evidence gate. The gate fired. See Why no router.

Setup & usability

Install, use, and (optionally) arm the guardrails

1 · Install

# add the marketplace, then install just this plugin
/plugin marketplace add abhijitbansal/claude-skills
/plugin install prompt-craft

2 · Use — explicit (pull)

/prompt-craft:improve-prompt add auth to the API
→ restated goal · acceptance criteria · assumptions to confirm
  · out of scope · recommended commands  (then it STOPS)

/prompt-craft:plan migrate the settings screen to the new design system

3 · Use — implicit (push / auto-invoke)

# no slash command — the lens fires from its description
"why is this test flaky?"        → debug lens
"clean this up, no behavior change" → refactor lens
"look this over before I push"     → review lens

4 · Arm the opt-in guardrails (off until you set them)

# shell profile, or .claude/settings.json "env" block
export PROMPT_CRAFT_BLOCK_SECRETS=1    # block reads/edits of secret-like files
export PROMPT_CRAFT_FORMAT_ON_EDIT=1   # auto-format edited files

The Stop-hook suggester is always on and always silent unless there's a concrete next step. No /init ceremony — two env vars are the entire surface (YAGNI).

Deep technical architecture

How it works under the hood

A · Where each hook fires in one turn

A "turn" is one user prompt → tool calls → stop. prompt-craft attaches at three lifecycle points. Hooks receive a JSON event on stdin and signal via exit code + stdout.

UserPromptSubmit
— (none)
No per-turn injection. The router that would live here was gated out — the platform's native description auto-invoke does this job.
PreToolUse
matcher: Read·Edit·Write
block_secrets.sh
Parses tool_input.file_path. If guardrail armed AND basename matches a secret pattern → exit 2 (blocks tool, stderr → Claude). Else exit 0.
PostToolUse
matcher: Edit·Write
format_on_edit.sh
If armed, dispatches the file's extension to an installed formatter (command -v guarded). Always exit 0 — never fails the edit.
Stop
suggest_next.sh
Reads cwd, inspects git: dirty → /commit+/review; clean+ahead-of-upstream → /pr; else silent. Emits hookSpecificOutput.additionalContext JSON.

Design rule (matches the repo's other hooks): non-guardrail hooks always exit 0 on the error/no-match path, so a hook fault can never block a user's prompt. Guardrails are the deliberate exception — they exit 2 to block.

D · The routing-spike scorer (the gate's engine)

A ~120-line stdlib tool (scripts/route_spike.py) that decides whether the router earns its existence. Two scorers over the real catalog:

# tokenize: lowercase, split on non-alnum, drop stopwords + len<3
score(prompt, entry, field) = | tokens(prompt) ∩ field_tokens(entry) |

keyword     field_tokens = tokens(entry.name)                  # the router's added signal
description field_tokens = tokens(name + entry.description)   # proxy for native auto-invoke

predict = argmax over catalog, or None if best < threshold(1)

Because any real router derives its keywords from the description, the description score is a ceiling the keyword router cannot beat. The spike just measures the gap on 40 labeled prompts.

Why intent-router was NOT built

The evidence gate fired

The original RFC proposed a maximal 4-tier router (keyword hook → cheap-model → MCP server + embeddings), a per-turn injection hook on by default, a generated-lens engine, and ~30 files. Before building any of it, Phase 2 ran a head-to-head on 40 labeled prompts against the 31-command real catalog:

Keyword/name router (the proposal)
50%
Description match (native, built-in, free)
70%
GATE: DO NOT build the router. Keyword routing loses to the platform baseline it would sit on top of by 20 points. It would add a maintained subsystem (catalog.json, freshness hook, per-turn injection, MCP server) to ship a weaker copy of a feature Claude Code already gives away for free. Paraphrased prompts ("save my work" → commit) are where keyword matching collapses and description matching wins.

What this bought: the high-value half (/improve-prompt, /plan, lenses, follow-ups) shipped immediately as prompt-craft; the speculative half was cut on evidence, not opinion. The 40-prompt labeled set survives as the regression backbone for /improve-prompt's recommended-commands. Cut to out-of-scope: T2 cheap-model tierT3 MCP + embeddingslens generatorcatalog auto-rewrite3 generic subagents
